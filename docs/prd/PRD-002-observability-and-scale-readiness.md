# PRD-002: Observability and Scale Readiness

## 1. Problem Statement
PRD-001@0.2.0 closed end-to-end (ARCH-001@0.4.0 + 15 implementation tickets merged) but no real-user 30-day pilot has yet started, so all PRD-001@0.2.0 §6 K1–K7 baselines remain `n/a`. Five operational lessons from the implementation pilots (BACKLOG-007 history, BACKLOG-008 summary, BACKLOG-009 deployment + PR-Agent CI tail-latency CRITICAL, BACKLOG-010 right-to-delete, BACKLOG-011 KPI smoke + Qwen 3.6 Plus 128K context exhaustion) surfaced four structural observability gaps: (1) per-role token / cost consumption is not measured, so silent overspend is invisible; (2) the tenant-isolation guarantee from PRD-001@0.2.0 §3 NG / G4 is verified only by an end-of-pilot audit, not continuously; (3) model-stall recovery (Qwen 3.6 Plus context exhaustion) was caught manually mid-pilot, with no automated detection; (4) the PR-Agent reviewer was swapped from Qwen 3.6 Plus to GPT-5.3 Codex on 2026-05-02 to mitigate the BACKLOG-009 5-of-5 cancellation pattern, but the swap has not yet been empirically validated. v0.2 also opens the user base from the 2-pilot population locked in PRD-001@0.2.0 §3 NG3 to a scale-ready growth path (Product Owner → partner → friends → eventually thousands), so the existing static Telegram allowlist (locked in ARCH-001@0.4.0 §3.1) and the PRD-001@0.2.0 §7 single-VPS resource ceiling no longer fit. PRD-002 closes those four observability gaps and establishes the scale-readiness foundation that the rest of the v0.2 roadmap (modalities expansion, proactive coaching + food guidance, calendar + web-view, life-manager modules) will be measured against; it ships before the modality and coaching epics so the next PRDs have honest per-role telemetry, continuous breach detection, and an empirically-validated PR-Agent runtime to build on.

## 2. Goals (SMART)
- **G1 — Per-role token-budget visibility.** Within 7 days of v0.2 deploy, every role in the model-assignment matrix locked on 2026-05-02 (Business Planner, Architect, Executor default + parallel + specialist, Reviewer auto + manual) has rolling-7-day, rolling-30-day, and current-day token consumption visible in a Product-Owner-readable dashboard at ≤15 minutes data-freshness, with at least 95% of routed calls attributed to the correct role. Measured server-side from structured-log emissions; baseline `n/a`, deadline measurement window opens at v0.2 deploy and persists indefinitely.
- **G2 — Continuous tenant-isolation breach detection.** From v0.2 deploy onward, every user-scoped storage operation that references a `user_id` other than the requesting principal emits a structured breach event within ≤5 minutes p95 of the operation, and a Telegram alert reaches the Product Owner within an additional ≤5 minutes. Over any rolling 30-day window the continuous detector must catch every breach the PRD-001@0.2.0 §6 K4 end-of-window audit catches (zero false-negatives). Measured by synthetic breach injection + structured-log timestamp diff. Promotes PRD-001@0.2.0 §6 K4 from end-of-pilot audit to continuous detection.
- **G3 — Automated model-stall detection.** From v0.2 deploy onward, any routed-model call (through the LLM-router layer or its direct-provider failover path) that produces no token output for ≥120 seconds (default; per-role configurable) emits a stall event within ≤15 seconds of the threshold being crossed, carrying enough context (call identifier, role, elapsed time, prompt-token count) for an operator to act on a takeover trigger. Coverage ≥95% of routed call paths; synthetic stall scenarios at 120 s / 300 s / 600 s all detected within target. Closes the manual-only recovery pattern documented in BACKLOG-011 §qwen-context.
- **G4 — Empirical PR-Agent tail-latency validation.** Within the first 10 pull requests merged after the PRD-002 telemetry deploys, PR-Agent CI exposes per-PR pre-token / first-token / last-token / total-duration metrics, the rolling-10-PR p100 PR-Agent CI total duration is ≤8 minutes (well below the 12-minute hard cancellation threshold observed in BACKLOG-009 §pr-agent-ci-tail-latency-investigation-CRITICAL on Qwen 3.6 Plus), and the rolling-10-PR p50 PR-Agent CI total duration is ≤4 minutes. Coverage = 100% of PRs from telemetry deploy onward. Empirically closes the 2026-05-02 GPT-5.3 Codex swap.
- **G5 — Scale-ready Telegram allowlist.** From v0.2 deploy onward, the Telegram access list is config-driven (no code change required to add or remove users) and passes load tests at N = 2, N = 10, N = 100, and N = 1 000 entries with allowlist-related per-update overhead ≤5 % of the user-facing latency budgets locked in PRD-001@0.2.0 §7 (voice ≤8 s p95, text ≤5 s p95, photo ≤12 s p95). Allowlist updates propagate from configuration to access enforcement within ≤30 seconds. Removes the 2-tenant ceiling implied by PRD-001@0.2.0 §3 NG3.
- **G6 — Per-call cost / token attribution.** From v0.2 deploy onward, ≥95 % of routed-model calls (through the LLM-router layer) and ≥95 % of direct-provider failover calls have per-call token-count and cost-estimate metadata captured in structured logs, with per-role rolling-7-day aggregates surfaced through the same dashboard as G1. Measured server-side; baseline `n/a` (PRD-001@0.2.0 had no per-role attribution).

## 3. Non-Goals (explicitly NOT in this epic)
- **NG1.** No new tracking modalities (no water / hydration, no sleep, no workout / exercise, no mood / energy logging). Deferred to the modalities-expansion PRD that follows PRD-002 in the v0.2 roadmap.
- **NG2.** No proactive nudges (missed-meal, under- or over-eating, macro-imbalance, pattern-triggered) and no adaptive-UX behaviour (assistant learning user preferences, adaptive questions, adjusting feature surfacing). Deferred to the proactive-coaching PRD.
- **NG3.** No food coaching beyond PRD-001@0.2.0 §3 NG6 (no vitamin / glycemic-index / supplement / hydration recommendations, on-request or unprompted). Deferred to the proactive-coaching PRD, which will define the new boundary against PRD-001@0.2.0 §3 NG7 (medical / clinical advice remains out of scope across the entire roadmap).
- **NG4.** No calendar integration of any kind (Google Calendar, Yandex Calendar, iCal, native calendars) and no web-based visualization layer (no read-only dashboards, no calendar-view web pages). Deferred to the calendar-and-web-view PRD.
- **NG5.** No per-user assistant personality customization (no preset picker, no tone-of-voice options). PRD-001@0.2.0 §3 NG10 is preserved for v0.2; the proactive-coaching PRD will revisit only the narrow adaptive-UX subset, not preset selection.
- **NG6.** No subscription, billing, payment, promo-code, or referral functionality. Extends PRD-001@0.2.0 §3 NG4 indefinitely; the Product Owner has confirmed monetization stays out of scope for the entire v0.2 cycle.
- **NG7.** No replacement of PRD-001@0.2.0 §5 user stories US-1 through US-9. The KBJU-only voice / text / photo logging surface, the daily / weekly / monthly summaries, the history mutation flow, the right-to-delete flow, and the multi-tenant data model remain in effect; PRD-002 only adds telemetry and scale around them, never replaces them.
- **NG8.** No medical or clinical advice. PRD-001@0.2.0 §3 NG7 is reaffirmed without modification; PRD-002 makes no statement about diagnoses, conditions, supplements, or pharmacology.

## 5. User Stories & Acceptance Criteria

### US-1: Per-role token-budget dashboard (G1, G6)
**As the Product Owner,** I want a single dashboard that shows, for every role in the model-assignment matrix, current-day / rolling-7-day / rolling-30-day token consumption and per-call cost estimates, **so that** I can spot a role drifting toward silent overspend before it crosses any budget threshold.
**Acceptance:**
- [ ] Given ≥7 days of telemetry after v0.2 deploy, when I open the dashboard, then I see per-role rolling-7-day token spend and rolling-7-day cost-estimate aggregates for every role active in the model matrix locked on 2026-05-02 (Business Planner, Architect, Executor default + parallel + specialist, Reviewer auto + manual).
- [ ] Given a role exceeds its per-role daily token-budget threshold (threshold value tracked in §9 OQ-3, ratified by the Product Owner), when the threshold is crossed, then a Telegram alert reaches the Product Owner within ≤5 minutes.
- [ ] Given a role is under threshold for a full rolling-7-day window, no false-positive alerts are emitted for that role across the same window.
- [ ] Given any routed-model call captured in the last 7 days, ≥95 % of those calls show per-call cost-estimate metadata in the dashboard.
- [ ] Dashboard data-freshness ≤15 minutes from event to display.

### US-2: Continuous tenant-isolation breach detection (G2)
**As the Product Owner,** I want every user-scoped storage operation that references a `user_id` other than the requesting principal to be detected and alerted within minutes, **so that** the tenant-isolation guarantee from PRD-001@0.2.0 §6 K4 is enforced continuously, not only at the end of a pilot window.
**Acceptance:**
- [ ] Given a user-scoped storage operation, when the operation references a `user_id` ≠ the requesting principal, then a structured breach event is emitted within ≤5 minutes p95 of the operation.
- [ ] Given a detected breach, when the event is emitted, then a Telegram alert reaches the Product Owner within an additional ≤5 minutes.
- [ ] Given any breach alert or breach log entry, no payload data, no `meal_text`, no usernames, no raw transcripts, no raw prompts, no raw media, no provider keys, and no provider responses appear (per ARCH-001@0.4.0 §10.7 emit-boundary redaction policy and §8.1 forbidden-field allowlist); only counts and tenant identifiers reduced to opaque IDs.
- [ ] Over any rolling 30-day window, the continuous detector catches every breach that the PRD-001@0.2.0 §6 K4 end-of-window audit (run via the dedicated audit role specified in ARCH-001@0.4.0 §9.2) catches: zero false-negatives.

### US-3: Automated model-stall detection (G3)
**As the Product Owner,** I want any LLM call that produces no token output for longer than a configurable threshold to be auto-detected and surfaced with takeover-trigger context, **so that** a stalled call (the BACKLOG-011 §qwen-context Qwen 3.6 Plus 128K context-exhaustion pattern) does not silently exhaust time and budget while I am away from chat.
**Acceptance:**
- [ ] Given an LLM call in flight, when no token output is observed for ≥120 seconds (default; per-role configurable), then a stall event is emitted within ≤15 seconds of the threshold being crossed.
- [ ] Given a stall event, the Product Owner receives a Telegram alert containing the call identifier, the role, the elapsed time, and the prompt-token count, with no raw prompt text included (subject to the ARCH-001@0.4.0 §8.1 redaction policy).
- [ ] The detector covers ≥95 % of routed-model call paths and ≥95 % of direct-provider failover paths captured by the runtime.
- [ ] Synthetic stall scenarios injected at 120 s, 300 s, and 600 s elapsed all trigger detection within the ≤15-second target.

### US-4: PR-Agent CI tail-latency telemetry (G4)
**As the Product Owner,** I want every merged pull request to expose per-PR pre-token / first-token / last-token / total-duration metrics for the PR-Agent CI stage, **so that** the 2026-05-02 GPT-5.3 Codex swap (research-PR + config-PR; rationale in `docs/knowledge/llm-model-evaluation-2026-05.md` §4 and §6 Q5) is empirically validated against the 5-of-5 cancellation baseline observed on Qwen 3.6 Plus in BACKLOG-009 §pr-agent-ci-tail-latency-investigation-CRITICAL.
**Acceptance:**
- [ ] Given a pull request that completed CI, when CI finished, then the PR-Agent stage exposes pre-token, first-token, last-token, and total-duration metrics tagged with the PR number, the commit SHA, and the runner identifier.
- [ ] Telemetry coverage = 100 % of pull requests from the deploy of PRD-002 telemetry forward.
- [ ] Rolling-10-PR p100 PR-Agent CI total duration ≤8 minutes (vs the 12-minute hard cancellation observed on Qwen 3.6 Plus in BACKLOG-009 §pr-agent-ci-tail-latency-investigation-CRITICAL).
- [ ] Rolling-10-PR p50 PR-Agent CI total duration ≤4 minutes.

### US-5: Scale-ready Telegram allowlist (G5)
**As the Product Owner,** I want to grow the user base from the current 2 tenants up to thousands by editing a configuration file (no code change, no redeploy), **so that** I can scale-test the v0.2 product gradually (Product Owner → partner → friends → wider population) without coupling user growth to engineering cycles.
**Acceptance:**
- [ ] Given a config-driven Telegram access list, when entries change from 2 to N, then no code deploy is required to enforce the new list.
- [ ] Given an updated access list, when configuration is committed, then access-control enforcement reflects the new list within ≤30 seconds.
- [ ] Allowlist load-tests pass at N = 2, N = 10, N = 100, and N = 1 000 entries.
- [ ] Per-update access-control overhead is ≤5 % of the user-facing latency budgets locked in PRD-001@0.2.0 §7 (voice ≤8 s p95, text ≤5 s p95, photo ≤12 s p95) at every load-test point.
- [ ] Removing a user from the access list immediately revokes access (no further user-scoped storage operations succeed for that `user_id` after revocation), and does not, by itself, delete that user's data; the PRD-001@0.2.0 §5 US-8 right-to-delete flow remains the only path that hard-deletes user data.

### US-6: Privacy-preserving telemetry (cross-cutting)
**As any user (Product Owner, partner, future user),** I want my data to remain redacted from every dashboard, alert, and structured-log channel that PRD-002 introduces, **so that** the new observability surface does not erode the data-isolation guarantee that PRD-001@0.2.0 §6 K4 promised.
**Acceptance:**
- [ ] Given any C10-emitted telemetry event introduced by PRD-002, no allowlisted forbidden field (`meal_text`, usernames, raw transcripts, raw prompts, raw media, provider keys, provider responses, per ARCH-001@0.4.0 §10.7 emit-boundary redaction policy and §8.1 allowlist) appears in dashboard, alert, or structured-log output channels.
- [ ] A sample audit of N ≥ 100 telemetry events drawn at random from the rolling-7-day window after v0.2 deploy shows 100 % redaction compliance.
- [ ] Any new telemetry event added by PRD-002 must declare which redaction-allowlist entries it asserts, and must be rejected at the emit boundary if it carries any non-allowlisted field.
- [ ] When a user invokes the PRD-001@0.2.0 §5 US-8 `/forget_me` flow, the right-to-delete transaction also hard-deletes that user's PRD-002 telemetry traces (per-call costs, per-role attribution rows, breach-detection events tagged with that `user_id`), in the same transaction boundary as the existing user-scoped data.

## 7. Technical Envelope (constraints Architect must respect)
- **Channel.** Telegram remains the primary product channel as locked in PRD-001@0.2.0 §7 (Product-Owner-locked decision). PRD-002 introduces no new user-facing channel; web visualization is explicitly deferred (§3 NG4) to a successor PRD in the v0.2 roadmap. Operator-facing dashboards introduced by PRD-002 are read by the Product Owner and the Architect (§4 P1, P4) and may be delivered through any operator-appropriate surface chosen by the Architect; they are not a user-facing channel.
- **Runtime.** openclaw runtime, locked in PRD-001@0.2.0 §7 and ARCH-001@0.4.0 (Product-Owner-locked decision). PRD-002 inherits this constraint without modification; the specific language and node version backing the runtime are Architect-owned and documented in ARCH-001@0.4.0.
- **VPS resource ceiling.** The PRD-001@0.2.0 §7 single-VPS ceiling (≤25 % CPU p95, ≤2 GB RAM steady) does NOT carry forward to v0.2 and beyond. PRD-002 sets no fixed CPU or RAM ceiling. The Architect must design for horizontal scale to ≥1 000 concurrent users (consistent with the §4 P3 growth path the Product Owner has confirmed: "это будет массовый продукт, разворачивать на тысячи юзеров"); concrete instance sizing is decided by the Architect and operations function on demand. Resource consumption should be proportional to active-user count, not unbounded.
- **LLM budget.** The PRD-001@0.2.0 §6 G5 / §7 $10/month total cost ceiling does NOT carry forward to v0.2; the Product Owner has explicitly deferred a v0.2 cost number ("сильно на это не опираемся, пока нам не важна стоимость в месяц"). PRD-002 commits no numeric LLM-budget envelope; the open value is tracked in §9 OQ-1. v0.2 implementation MUST still emit per-role and per-call cost telemetry (G1 / G6) so that the Product Owner can ratify a numeric envelope as soon as one is required for production scale.
- **Latency budget.** PRD-002 telemetry collection MUST add ≤5 % overhead to every user-facing latency budget already locked in PRD-001@0.2.0 §7 (voice ≤8 s p95 / ≤30 s p100; text ≤5 s p95 / ≤15 s p100; photo ≤12 s p95 / ≤45 s p100). PRD-002 introduces three new operator-facing latency targets: K1 dashboard data-freshness ≤15 min, K2 breach-detection p95 ≤5 min, K3 stall-detection emission ≤15 s after threshold cross.
- **Localization.** Russian-only user-facing UX, unchanged from PRD-001@0.2.0 §7. Operator-facing dashboards and alerts may be in Russian or English at the Architect's discretion; user-facing surfaces (Telegram replies in PRD-001@0.2.0 §5 user stories) are not affected by PRD-002.
- **Compliance and PII handling.** PRD-002 telemetry channels (dashboards, alerts, structured logs introduced by this PRD) MUST honour the redaction allowlist locked in ARCH-001@0.4.0 §8.1 and the emit-boundary redaction policy locked in ARCH-001@0.4.0 §10.7 (TKT-015@0.1.0 hardening). No new redaction-policy weakening is permitted by PRD-002. Telegram Bot API Terms of Service (`https://core.telegram.org/bots/terms`) compliance is unchanged from PRD-001@0.2.0 §7.
- **Data retention.** User-facing data-retention rules from PRD-001@0.2.0 §7 (raw audio and raw photo bytes deleted immediately after extraction; transcripts and confirmed meals retained until right-to-delete) carry forward unchanged. PRD-002 adds telemetry data (per-role aggregates, breach-detection events, model-stall events, PR-Agent CI metrics) whose retention duration is open and tracked in §9 OQ-2.
- **Right-to-delete scope extension.** PRD-001@0.2.0 §5 US-8 right-to-delete is extended in PRD-002 §5 US-6 to also hard-delete the requesting user's PRD-002 telemetry traces (per-call cost rows, per-role attribution rows, breach-detection events tagged with that `user_id`) within the same transaction boundary as the existing user-scoped deletion in ARCH-001@0.4.0 §3.11.
- **External dependencies.** Telegram Bot API; openclaw runtime; the LLM-router layer with direct-provider failover keys (per `docs/knowledge/llm-routing.md`). PRD-002 introduces no new external dependency. No web service, no external calendar provider, no new vendor lock-in is added by this PRD.

## 8. Risks & Mitigations
| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| **R1 — telemetry collection adds latency that breaches §7 user-facing budgets.** | high | medium | The Architect designs telemetry as asynchronous emit-and-forget against the existing C10 path; sampled metrics where full-volume telemetry would breach the ≤5 % overhead constraint; an explicit load-test gate in the implementation tickets blocks merge if the budget is violated. |
| **R2 — GPT-5.3 Codex itself stalls or runs long, so the PR-Agent swap does not actually solve BACKLOG-009.** | high | low–medium | G4 forces empirical validation across the first 10 PRs; if rolling-10-PR p100 exceeds 8 minutes, the Devin Orchestrator escalates to the Product Owner per the BACKLOG-009 closure protocol, and a fallback model is declared by the Architect in a successor ADR. The 2026-05-02 swap is treated as a hypothesis until G4 confirms it. |
| **R3 — the breach detector becomes a cross-tenant query path itself.** | critical | low | The detector runs through the dedicated audit role gated by a separate runtime secret per ARCH-001@0.4.0 §9.1 / §9.2; it returns counts only, no payload; the Reviewer audits the detector implementation in the same way it audited the end-of-pilot K4 audit query. |
| **R4 — allowlist scaling reveals a per-update bottleneck (e.g. linear scan on every Telegram update).** | medium | medium | The Architect designs cached lookup with a published cache-invalidation contract; load-tests at N = 10 / 100 / 1 000 are gates in the implementation tickets, not after-the-fact validation; if any gate fails, the Architect redesigns before user growth past that gate is enabled. |
| **R5 — per-role cost / token attribution is partially missed because C10 today (per ARCH-001@0.4.0 §3.10) covers per-call spend but not per-role aggregation across the new 2026-05-02 model matrix.** | medium | medium | The Architect extends C10 in the successor ArchSpec; per-role attribution is declared in skill manifests at design time, not inferred at runtime; the G1 / G6 acceptance threshold of ≥95 % attribution is a hard gate. |
| **R6 — PRD-002 telemetry leaks PII (`meal_text`, usernames, raw transcripts, raw prompts, raw media) into dashboards or alerts.** | critical | low | PRD-002 inherits the ARCH-001@0.4.0 §10.7 emit-boundary redaction policy; US-6 acceptance includes a 100-event sample audit; any new event must declare its allowlisted fields and is rejected at the emit boundary if it carries non-allowlisted fields. |
| **R7 (weakest assumption A1) — the 2026-05-02 GPT-5.3 Codex swap empirically resolves the BACKLOG-009 5-of-5 cancellation pattern; this has not been verified yet, only inferred from offline benchmarks.** | high | medium | G4 is the validation; the PRD itself does not assume the swap works, it forces measurement. If the assumption fails, R2's escalation path applies. |
| **R8 (weakest assumption A2) — continuous tenant-breach detection can run inside the existing C10 telemetry stack within the §7 telemetry-overhead constraint, without requiring a parallel detector pipeline.** | medium | medium | The Architect performs a feasibility study before committing detector design; if the constraint cannot be met inside C10, sampling fallback (≤1 in N persistence operations checked synchronously, the rest checked asynchronously in a parallel queue) is acceptable, with the sampling rate set to recover G2's zero-false-negative target on a rolling 30-day basis. |
| **R9 (weakest assumption A3) — Telegram allowlist scaling to ≥1 000 entries works through configuration rotation alone, with no deeper code-path bottleneck (e.g. Telegram update handler doing linear scan, per-message access check round-tripping a slow store).** | medium | medium–high | TKT-level load-tests at N = 10 / 100 / 1 000 gate merge; if a code-path bottleneck is found, a new caching component is required and the Product Owner is notified that the user-count growth path must pause at the last-passed gate until the bottleneck is resolved. |

## 9. Open Questions (resolve BEFORE handoff to Architect)
- **OQ-1 — v0.2 LLM cost envelope.** The Product Owner has explicitly deferred a v0.2 cost number for now ("сильно на это не опираемся, пока нам не важна стоимость в месяц"). Before the next epic in the v0.2 roadmap (modalities expansion) freezes its own §7, the Product Owner must ratify either a per-user $/month cap, a total $/month cap with tiered alerts, or an explicit "no enforced cap; track only" stance. Default if unset at the PRD-002-merge moment: track per role (G1 / G6) but do not enforce a numeric ceiling, and surface this as a §9 OQ in the next PRD too.
- **OQ-2 — telemetry data retention duration.** PRD-002 introduces per-role aggregates, breach-detection events, model-stall events, and PR-Agent CI metrics whose retention duration is not specified. The Product Owner must ratify either a fixed window (e.g. 30 / 90 / 365 days) or a tiered policy (live store + cold archive + auto-purge). Default if unset before Architect handoff: 90-day live retention, then archive, then auto-purge at 365 days.
- **OQ-3 — numeric thresholds for K1 / K2 / K3.** The K1 alert threshold (per-role daily token-budget percentage that triggers a Telegram alert), the K2 breach-detection emission target latency in synchronous vs sampling configurations, and the K3 stall threshold (default 120 seconds; per-role configurable) all require Architect feasibility input followed by Product Owner ratification.
