# Pilot KPI Smoke Suite follow-ups + fifth TO pilot lessons

This file collects the structural lessons from the TKT-014 closure (PR #89 Executor + PR #90 Reviewer + closure-PR #91), the fifth and final end-to-end Ticket Orchestrator pilot in the 15-TKT KBJU Coach v0.1 pipeline. The first pilot (TKT-010) generated `BACKLOG-007` and `PR #71`; the second (TKT-011) generated `BACKLOG-008`; the third (TKT-013) generated `BACKLOG-009`; the fourth (TKT-012) generated `BACKLOG-010`; this fifth and final pilot generates `BACKLOG-011`.

The 5 entries below are all **structural TKT-NEW** entries arising from anomalies surfaced during the TKT-014 cycle. They split into:

- **2× executor-runtime structural** (Qwen 3.6 Plus 128k context insufficient as Executor for repository-context tickets; Executor opencode session lacks git push credentials).
- **1× executor-takeover protocol** (mid-cycle Executor handover from Qwen to Codex clobbered the TKT frontmatter `status` field).
- **2× pipeline-improvement structural** (TO-NUDGE files MUST be valid CommonMark/GFM; formal TO → Devin Orchestrator postmortem feedback loop in hand-back format).

Zero deferred Reviewer / PR-Agent code findings — all 7 Kimi findings (F-H1 through F-L3) and all 6 PR-Agent final-head findings (K1 vacuous pass / Cyrillic homoglyph redaction / unused `redactK1Report` / K7 daily macro tolerance / K2 order dependency / K4 latest audit run) were RESOLVED on Executor PR #89 final HEAD `3c6ff96`. This is the best procedural discipline of all 5 TO pilots (TKT-013 deferred 17, TKT-012 deferred 2, TKT-010/011 deferred 0/4 respectively) and reflects accumulated process maturity from BACKLOG-007/008/009/010 lessons.

## TKT-NEW-qwen-3.6-plus-128k-context-insufficient-for-executor

**Source:** TKT-014 cycle empirical observation. PO dispatched Executor opencode session with Qwen 3.6 Plus via OmniRoute per Architect-original frontmatter assignment + Variant A election (honour Architect plan, completing the 3-family executor-uncorrelation calibration matrix). Qwen iter-1 implementation completed successfully (5m58s, commit `c1c97f2` — 18/18 tests, lint, typecheck, validate_docs all PASS); Qwen iter-2 (responding to Kimi RV-CODE-014 iter-1 findings F-H1 + F-M1 + F-M2 + F-M3 + F-L1 + F-L2 + F-L3) **stalled on context exhaustion**. PO direct quote (chat 2026-05-02): «Qwen не справлялась совсем (контекст всего 128к, она тупила и тд, невозможно было закончить)» — translation: "Qwen didn't cope at all (context only 128k, it was sluggish, etc., impossible to finish)". PO mid-cycle switched to Codex GPT-5.5 high via OmniRoute from iter-2 onward; Codex completed iter-2 + iter-3 + iter-4 + iter-5 successfully on the final HEAD `3c6ff96`. The Architect-original 3-family matrix (3 GLM 5.1 + 1 Codex GPT-5.5 + 1 Qwen 3.6 Plus across 5 pilots) thus DEGRADED to 3 GLM + 2 Codex; **Qwen 3.6 Plus remains UNVALIDATED as Executor across the entire 15-TKT pipeline** (TKT-011 was also originally Qwen-assigned but ran on GLM 5.1 — see `BACKLOG-008 §launcher-asserts-frontmatter-executor` and PR #77 retroactive frontmatter bump).

**The issue:** Qwen 3.6 Plus advertises a 128k-token context window. For TKT-014's task surface — reading 5 production source files (`src/store/types.ts`, `src/history/historyService.ts`, `src/privacy/rightToDelete.ts`, `src/summary/recommendationGuard.ts`, `src/photo/photoConfidence.ts`, `src/privacy/tenantAudit.ts`), 7 ArchSpec / ADR pinned references, the TKT body itself, the cumulative diff history, the Reviewer findings (Kimi RV-CODE-014 iter-1 review at >700 lines), and producing 5 implementation files (kpiQueries.ts, pilotReadinessReport.ts, fixtures.ts, kpiQueries.test.ts, pilotSmoke.test.ts) — the cumulative iter-2 prompt size empirically exceeded Qwen's effective working window. The model became "sluggish" (high time-per-token), output low-quality continuations, and eventually could not produce a coherent fix for F-H1 (the largest finding, mandating a complete pilotSmoke.test.ts rewrite with production imports). This is not a transient throughput issue — Qwen iter-1 (smaller cumulative prompt) succeeded at 5m58s; Qwen iter-2 (larger cumulative prompt with Reviewer findings + iter-1 code-state + production-source-file context) failed.

**Concrete pipeline impact:** With BACKLOG-008 §launcher-asserts-frontmatter-executor codified, post-hoc frontmatter bumps are the documented escape hatch when actual ≠ assigned executor. But the deeper question is **whether Qwen 3.6 Plus is a viable Executor model at all for any TKT requiring multi-file repository context**. The 15-TKT pipeline never produced a successful end-to-end Executor cycle on Qwen; both Qwen-assigned tickets (TKT-011 + TKT-014) ran on substitutes (GLM 5.1 and Codex GPT-5.5 respectively). The Architect-rationale on TKT-014 §7 ("Qwen assignment is appropriate because this ticket is parallel-review friendly and focused on integration evidence across completed modules") was reasonable in theory but did not account for the cumulative prompt-size growth across iter-N — the model is loaded with the entire repo-bootstrap context PLUS the cumulative review findings PLUS its own previous iteration's code-state.

**Proposed fix:**

1. **Strategic (separate research-PR — feeds the next entry):** PO requested a model re-evaluation across all roles and all available models (GLM 5.1 / DeepSeek-V4-Pro / Qwen3.6 Plus / Kimi K2.6 / MiniMax M2.7 / GPT-5.5 family) using arena.ai / lmarena / livebench evidence. Specifically for Executor: candidate replacements for the "parallel" slot include DeepSeek-V4-Pro and MiniMax M2.7 if their effective context windows exceed 200k+ tokens at usable throughput; if no Fireworks alternative offers sufficient context, the parallel slot may be dropped entirely (collapsing Executor to default GLM 5.1 + specialist Codex GPT-5.5 only — 2 families instead of 3). Cost-aware: GPT-5.5 budget is finite per PO; Fireworks alternatives are unlimited per PO. The research-PR is a successor to this entry and will resolve it.

   **Update 2026-05-02 (pt2 — research note opened in PR #93):** `docs/knowledge/llm-model-evaluation-2026-05.md` recommends **Qwen 3.6 Plus → DeepSeek V4 Pro Executor-parallel swap** (DeepSeek 1M context, BenchLM Coding 73.8 vs Qwen 64.8, Knowledge 62.6 highest of all Fireworks-hosted candidates, $1.74/$3.48 pricing). Restores 3-family Executor uncorrelation matrix (GLM 5.1 + DeepSeek V4 Pro + Codex GPT-5.5) per CONTRIBUTING.md §7.

   **Update 2026-05-02 (pt3 — PO Q1 approved + Q6 verified):** PO accepted DeepSeek V4 Pro = Executor parallel (Q1 chat 2026-05-02: «да, делаем дипсик 4 про параллельным»). ## TKT-NEW-executor-cannot-push-without-auth

**Source:** TKT-014 iter-1 + iter-2 empirical observation. Qwen Executor session attempted `git push -u origin tkt/TKT-014-pilot-kpi-smoke-suite` and received HTTP 403: `remote: Permission to OpenClown-bot/openclown-assistant.git denied to OpenClown-bot. fatal: unable to access 'https://github.com/OpenClown-bot/openclown-assistant.git/': The requested URL returned error: 403`. PO direct quote (chat 2026-05-02): "ранее и от других экзекуторов, я получаю это" — confirming this is a portable / repeatable failure mode, not specific to Qwen. PO had to manually run the push commands from a credentialed shell.

**The issue:** The PO's Windows-PC opencode session — used for ALL Executor model variants (GLM 5.1, Qwen 3.6 Plus, Codex GPT-5.5) and for the Ticket Orchestrator session — does not currently have a configured git credential helper / GitHub PAT (Personal Access Token) authenticated against the `OpenClown-bot/openclown-assistant` repository. The session can clone (read access via HTTPS) but cannot push (write access requires PAT or SSH key). When the Executor model emits the canonical `git push -u origin <branch>` command as part of its TKT-cycle hand-off, the command silently fails at the transport layer. The Executor model may then waste tool calls retrying or generating fallback diagnostics; the actual fix is operational (PO sets up credentials), not in the Executor model's logic.

**Concrete pipeline impact:** This is a procedural friction tax on every Executor cycle. PO has to manually intervene at the push step, which (a) breaks the closed-loop "Executor commits → Reviewer reviews → cycle iterates" model that the TO bootstrap assumes, and (b) creates surface area for the BACKLOG-011 §mid-cycle-clobber failure mode below (Executor takes over from a partial / unpushed state and may inadvertently overwrite frontmatter set by the prior Executor's local-only commits).

**Proposed fix:**

1. **One-time PO setup (one-shot operational fix):** PO configures a GitHub PAT scoped to `repo` permissions on `OpenClown-bot/openclown-assistant` and registers it via `git config --global credential.helper` (Windows Credential Manager) or via `gh auth login` if `gh` is installed. PO verifies by running `git push -u origin <test-branch>` from the opencode session terminal before any future TKT dispatch.

2. **TO bootstrap addition (process fix):** Add a new pre-flight check in the TO bootstrap (and in the Devin Orchestrator → TO bootstrap delivery message) requiring PO to confirm Executor opencode session has git push credentials BEFORE iter-1 dispatch. The TO must explicitly ask: "Confirm that `git push -u origin <test-branch>` works from the Executor opencode session shell." If PO confirms, proceed. If not, PO sets up credentials before iter-1.

3. **Documentation update (informational):** Add a one-paragraph note to `docs/meta/devin-session-handoff.md` § Delegating to Ticket Orchestrator describing the auth requirement.

**Severity:** Medium (procedural friction; does not affect code correctness; affects pipeline velocity and creates surface area for state-clobber failure modes). Resolution is one-time PO config; the TO bootstrap fix can be applied immediately in the TO prompt update.

**ArchSpec dependency:** None — this is operational / infrastructure, not code/ArchSpec.

**Status:** Open. PO to configure credentials; TO bootstrap update lands in this closure-PR's `docs/meta/devin-session-handoff.md` §11 update.

## TKT-NEW-mid-cycle-executor-takeover-clobbers-frontmatter-status

**Source:** TKT-014 cycle empirical observation. Qwen iter-2 commit `91d4718` (a clerical commit titled "TKT-014: flip status in_progress -> in_review, fill execution log") correctly set TKT-014 frontmatter `status: in_review`. After mid-cycle Qwen → Codex GPT-5.5 high takeover at iter-2, Codex's iter-3 commit `7dc4606` (a substantive commit titled "TKT-014: iter-3 fix K1 vacuous pass + rewrite pilotSmoke with production exports") **regressed the frontmatter status from `in_review` back to `ready`**. The status remained `ready` for the rest of the cycle (commits `29f2fd4` iter-4, `3c6ff96` iter-5 final HEAD). Final HEAD merged with frontmatter `status: ready` despite the cycle being in active code review. Discovered during Devin Orchestrator pass-2 ratification audit by reading frontmatter at each commit hash; neither TO pass-1, Kimi K2.6, nor PR-Agent (Qwen 3.6 Plus) flagged the regression.

**The issue:** When Codex took over from Qwen mid-cycle, the takeover protocol per the §10 Execution Log was: «Codex GPT-5.5 high via OmniRoute: iter-2 takeover after Qwen stalled; reset local undelivered partial work to origin/PR HEAD 91d4718». The reset target `91d4718` correctly had `status: in_review`. However, when Codex started fresh in a new opencode session and re-read the TKT body to refresh its working context, it appears to have read an OLDER snapshot of the file (perhaps from the `tkt/TKT-014-pilot-kpi-smoke-suite` branch's pre-flip state, or from a stale local cache) that still showed `status: ready`. When Codex then re-saved the file as part of the iter-3 substantive fix (which legitimately needed to edit §10 Execution Log), the save operation re-wrote the entire frontmatter from Codex's stale snapshot, overwriting Qwen's `status: in_review` flip with the older `status: ready`. None of the three reviewers (TO, Kimi, PR-Agent) had a check that compared the final HEAD frontmatter `status` field against the expected `in_review` (or `done` post-merge) values. The `validate_docs.py` script enforces frontmatter schema (allowed values for `status`) but not state-machine transitions (forbidden transitions like `in_review → ready` mid-cycle).

**Concrete pipeline impact:** The merge proceeded with stale frontmatter, and the Devin Orchestrator closure-PR had to flip `ready → done` directly (skipping the intermediate `in_review` state that was set then clobbered). This is a procedural inconsistency that is not caught by current automated checks; an attentive Devin Orchestrator caught it in pass-2, but a less rigorous pass would have missed it (the visible commit titles say "flip status to in_review", which gives a false sense that the field is in the expected state without actually reading the final HEAD frontmatter). Also note: TKT-014 had FIVE Executor iterations because of the Qwen → Codex takeover; the regression introduced at iter-3 persisted across two more substantive iterations + two further reviewer rounds. The regression window was 4+ hours wide.

**Proposed fix:**

1. **Reviewer prompt update:** Add a hard check to `docs/prompts/reviewer.md` Reviewer contract: "On final HEAD, verify TKT frontmatter `status: in_review` (or expected closure state). If status field does not match the expected value, flag as Reviewer finding and require Executor frontmatter restoration BEFORE pass verdict." This catches the regression at the second-line defense.

2. **TO pass-1 audit update:** Add a hard check to TO pass-1 cross-reviewer audit checklist (in `docs/prompts/ticket-orchestrator.md` and the cross-referenced `docs/meta/devin-session-handoff.md` §11.3 hand-back format): "Verify TKT frontmatter `status` field at final HEAD matches the expected mid-cycle value (`in_review` for code-review-in-progress, `done` only after closure-PR). Mismatch is a hand-back blocker — TO must require Executor to push a clerical fix commit before hand-back."

3. **`validate_docs.py` extension (deferred to a future TKT-NEW; out of scope for closure-PR #91):** Add a state-machine check to `validate_docs.py` that asserts (a) the final commit on a TKT branch has `status` matching the PR's review state, and (b) consecutive commits on the same branch do not regress the `status` field (e.g., `in_review → ready` is forbidden, `ready → in_progress → in_review → done` is the only allowed forward path). Defer this to BACKLOG-011 follow-up because it requires git-walk logic that current `validate_docs.py` does not have.

4. **TO bootstrap addition (mid-cycle takeover protocol):** Add an explicit "mid-cycle takeover" sub-protocol to `docs/meta/devin-session-handoff.md` §11.X: when PO mid-cycle switches Executor models (e.g. Qwen stalled → Codex takes over), the takeover Executor MUST first run `git fetch origin` + `git checkout <pr-branch>` + `git pull --ff-only` to ensure local state matches origin HEAD before any new write operation. Avoid `git reset --hard origin/<branch>` because that loses local commits if the prior Executor pushed any (which Qwen had — the iter-2 status-flip commit `91d4718`). The takeover Executor MUST also explicitly read the post-pull frontmatter to confirm state before editing.

**Severity:** Low (single-pilot anomaly; no AC violation in TKT-014 because the §10 Execution Log was correctly filled and the closure-PR fixes the status). Production correctness unaffected — TKT frontmatter `status` is metadata only, not code behavior. But the failure mode is portable (any Executor takeover scenario can hit it) and the catch-rate is currently 0/3 reviewers, which is an acceptable-quality-bar issue at the procedural layer.

**ArchSpec dependency:** None — this is process/prompt-level, not code/ArchSpec.

**Status:** Open. Reviewer prompt update + TO pass-1 audit update + mid-cycle takeover sub-protocol all land in this closure-PR's `docs/prompts/ticket-orchestrator.md` + `docs/meta/devin-session-handoff.md` updates (out-of-DO-write-zone files; PO authorisation recorded verbatim in PR body). `validate_docs.py` state-machine extension deferred to future BACKLOG entry.

## TKT-NEW-to-nudge-must-be-valid-markdown-formatting

**Source:** PO Meta-feedback #3 (chat 2026-05-02 with attached screenshot of Reviewer NUDGE rendered as plain text in opencode session). PO direct quote: «я думаю, что нужно строго в маркдауне писать промпты для экзекуторов и ревьюверов, потому что сейчас это выглядит криво (ТО должен так писать для них промпты)» — translation: "I think we need to strictly write prompts for Executors and Reviewers in Markdown, because right now it looks ugly (TO must write prompts for them this way)".

**The issue:** The current `docs/prompts/ticket-orchestrator.md` instructs the TO to "draft NUDGE files (paste-text, not committed)" for Executor and Reviewer dispatch. The format is informal — TOs (running on GPT-5.5 thinking) have been writing NUDGE bodies with mixed conventions: some use `<<<>>>` markers as section delimiters, some use plain ASCII headings, some use proper Markdown. When PO pastes these NUDGE bodies into the Executor / Reviewer opencode session as the first user message, the opencode chat UI renders them as Markdown by default. NUDGE files written with non-Markdown markers (e.g. `<<<HEADING>>>` instead of `## HEADING`, plain dashes instead of code-fences for command blocks, no `**bold**` or `_italic_` for emphasis) render as a wall of unstructured plain text — visually hard to parse, easy to miss key constraints, and undermines the Executor / Reviewer's ability to extract the contract surface efficiently. The screenshot PO provided shows a TKT-014 Reviewer NUDGE rendered as plain text with no visible structure; PO flagged this as a process-quality issue.

**Concrete pipeline impact:** Sub-optimal NUDGE rendering increases the risk of Executor / Reviewer missing constraints (e.g., the canonical-filename requirement, the iter-N continuation rule, the Reviewer-re-engagement constraint, the Architect-rationale citation). The mitigation has been "TO writes longer NUDGE bodies with extra repetition", which (a) wastes context tokens, and (b) is a workaround for a formatting issue that should be solved at the source.

**Proposed fix:**

1. **`docs/prompts/ticket-orchestrator.md` update:** Add a new mandatory sub-section "TO-NUDGE Formatting Contract" specifying: (a) NUDGE bodies MUST be valid CommonMark / GitHub Flavored Markdown — `## H2` for major sections, `### H3` for sub-sections, ` ``` ` triple-backtick code-fences for command blocks and contract excerpts (with optional language tag for syntax highlighting), `-` or `*` for bulleted lists, `1.` for numbered lists, `**bold**` for emphasis on critical constraints, `_italic_` for citations or sub-emphasis, blockquote `>` for quoted ArchSpec / TKT excerpts. (b) NUDGE bodies MUST NOT use ad-hoc markers like `<<<HEADING>>>`, `==== SECTION ====`, or other non-Markdown-rendering conventions. (c) NUDGE bodies SHOULD include a fenced code-block at the top with `markdown` language tag containing a one-line "Iter-N <ROLE> for TKT-NNN" header that the Executor / Reviewer can immediately recognize as the dispatch boundary. (d) NUDGE bodies MUST end with the explicit "Wait for me ('go') before you ..." gate so the Executor / Reviewer holds before any write.

2. **`CONTRIBUTING.md` Hard rule update:** Add a new Hard rule (after the existing § Hard rules section): "TO-NUDGE files (paste-text dispatched into Executor / Reviewer opencode sessions) MUST be valid CommonMark / GitHub Flavored Markdown. Non-Markdown ad-hoc markers (e.g. `<<<...>>>`, `=== ... ===`) are forbidden because they render as unstructured plain text in the opencode chat UI and undermine Executor / Reviewer comprehension. See `docs/prompts/ticket-orchestrator.md` § TO-NUDGE Formatting Contract." This makes the rule discoverable from the top-level repo docs even without reading the TO prompt.

3. **Devin Orchestrator → TO bootstrap delivery template update:** The Devin Orchestrator's TO-bootstrap delivery message itself is already in valid Markdown (this closure-PR continues that pattern). No change needed there; the rule applies to the TO's own NUDGE outputs to Executor / Reviewer.

**Severity:** Medium (process / quality friction; affects every Executor and Reviewer dispatch across all future TKT cycles; resolution is a one-time prompt update). Resolution lands in this closure-PR.

**ArchSpec dependency:** None — this is process / prompt-level, not code/ArchSpec.

**Status:** Open. Resolution = `docs/prompts/ticket-orchestrator.md` § TO-NUDGE Formatting Contract addition + `CONTRIBUTING.md` Hard rule addition, both landing in this closure-PR with PO authorisation recorded verbatim per `CONTRIBUTING.md` row 23.

## TKT-NEW-to-do-postmortem-loop-formal-channel

**Source:** PO Meta-feedback #2 (chat 2026-05-02). PO direct quote: «и в целом, сделай (если еще мы это не реализовали) передачу информации от ТО к тебе. то есть, если что-то у ТО происходит в процессе - он тебе потом это передает как опыт, а ты его изучаешь и, если надо, добавляешь себе(нам в репо)» — translation: "And in general, set up (if we haven't already implemented this) the transfer of information from TO to you. That is, if something happens with the TO during the process — he then passes that to you as experience, and you study it and, if necessary, add it to yourself (to the repo for us)".

**The issue:** The current `docs/meta/devin-session-handoff.md` §11.3 hand-back format defines what the TO must deliver to the Devin Orchestrator at cycle close: PR links, final HEADs, Reviewer verdict + iteration count, PR-Agent settle status, deferred findings list, recommendation. It does NOT define a section for "operational lessons learned this cycle" — i.e., procedural insights the TO observed during the cycle that are not Reviewer findings (which target the code) but are pipeline observations (e.g., "Qwen 128k context insufficient; PO had to mid-cycle switch", "push 403 fail on iter-1 due to opencode session lacking PAT", "Codex iter-3 clobbered frontmatter status when reading stale local snapshot"). Currently these observations reach the Devin Orchestrator only through PO mediation (as happened on TKT-014: PO chat 2026-05-02 contained the operational notes verbatim because the TO had already concluded its session and the only path back to the Devin Orchestrator was through PO).

This is a one-directional pipeline (Devin Orchestrator → TO bootstrap → TO executes → TO hand-back → Devin Orchestrator pass-2) with operational learnings either lost or routed through PO out-of-band. PO correctly flagged this as a closed-loop deficit.

**Concrete pipeline impact:** Three of the BACKLOG-011 entries above (qwen-context-fail, push-auth-fail, mid-cycle-clobber) reached the Devin Orchestrator through PO chat rather than through TO hand-back — slowing closure-PR drafting (Devin Orchestrator had to re-discover the issues from PO chat fragments rather than reading them as a structured TO-authored section). Future TO pilots may surface similar operational observations that should be codified into BACKLOG entries during pass-2; without a formal hand-back section, these observations are at risk of being lost.

**Proposed fix:**

1. **`docs/meta/devin-session-handoff.md` §11.3 hand-back format update:** Add a new mandatory section "TO Operational Notes" to the TO hand-back format. Specification:
   - **Section name:** `## TO Operational Notes (for Devin Orchestrator postmortem capture)`
   - **Purpose:** Capture procedural insights observed during the cycle that are NOT Reviewer findings on the code but are pipeline / process observations.
   - **Structure:** Free-form bullet list. Each bullet should describe ONE observation in 1-3 sentences with: (a) what happened, (b) why it happened (best understanding), (c) the impact on the cycle, (d) whether the TO worked around it or was blocked.
   - **Examples of what belongs here:** Executor model failure modes (context exhaustion, throughput issues, push auth fails, takeover clobbers); opencode session interruptions; PR-Agent CI behavior anomalies; cross-reviewer audit difficulties; unexpected ArchSpec / TKT contradictions discovered mid-cycle; PO-mediated decisions and their impact.
   - **Examples of what does NOT belong here:** Reviewer code findings (those go in RV-CODE artifact + per-finding resolution table); deferred follow-ups (those go in BACKLOG via closure-PR); routine cycle metrics (those go in TKT §10 Execution Log).
   - **Integration with pass-2:** Devin Orchestrator pass-2 ratification MUST read the TO Operational Notes and decide for each bullet: (a) codify as a new BACKLOG TKT-NEW entry (structural / portable lesson), (b) note in session-log only (single-pilot anomaly, archive), or (c) escalate to PO via chat (urgent / requires PO input). The decision is recorded in the closure-PR body and/or session-log.

2. **`docs/prompts/ticket-orchestrator.md` update:** Add a corresponding § "Operational Notes capture protocol" describing what the TO should and should not write in the TO Operational Notes section, with examples drawn from BACKLOG-007/008/009/010/011.

3. **First application:** This very closure-PR demonstrates the pattern in retrospect — three BACKLOG-011 entries (qwen-context, push-auth, mid-cycle-clobber) are exactly the kind of observations that would have gone into TO Operational Notes had the format existed. The Devin Orchestrator captured them from PO chat instead.

**Severity:** Medium (process / closed-loop completeness; affects every future TKT cycle's hand-back quality; resolution is a one-time prompt + handoff-doc update). Resolution lands in this closure-PR.

**ArchSpec dependency:** None — this is process / prompt-level, not code/ArchSpec.

**Status:** Open. Resolution = `docs/meta/devin-session-handoff.md` §11.3 update + `docs/prompts/ticket-orchestrator.md` § Operational Notes capture protocol addition, both landing in this closure-PR with PO authorisation recorded verbatim per `CONTRIBUTING.md` row 23.

## TKT-NEW-mocked-end-to-end-smoke-tests-false-confidence-process-retro

**The issue:** Five Ticket Orchestrator pilots (TKT-010 through TKT-014) closed end-to-end with all required gates green: Reviewer Kimi K2.6 + Qodo PR-Agent + Ticket Orchestrator audit pass-1 + Devin Orchestrator ratification audit pass-2 + `python3 scripts/validate_docs.py` + Vitest unit tests + `tsc --noEmit` lint/typecheck. At least one of the 5 pilots (TKT-014 Pilot KPI Smoke Suite) was explicitly intended to deliver an end-to-end smoke harness that would catch deployment-time regressions. None of the 15+ closure-audit gates flagged the absence of a runnable boot path on `main`, because the pilot-readiness smoke suite (`src/pilot/pilotReadinessReport.ts`, `tests/pilot/*.test.ts`) is **mocked at every external boundary** — mock Telegram client, mock OmniRoute provider, mock Postgres pool, mock OpenClaw context — and asserts only that the **modules** imported from `dist/...` behave correctly under those mocks. There is no test (or CI step, or runbook step) that builds a container from `Dockerfile`, starts it via `docker compose up`, and asserts the process stays alive long enough to receive at least one external signal (e.g. a `getMe` round-trip on the Telegram API, or a structured `[gateway] ready` log line).

This is a **structural false-confidence pattern**: «green CI + green Reviewer + green TO + green DO ratification» does not imply «v0.1 product is deployable», because the audit surface and the deploy surface intersect only at the file-existence and type-correctness layer, not at the process-startup or runtime-integration layer.

**Proposed fix shape (Architect Phase-0 Recon to formalise in ARCH-001 v0.5.0 §11 Test Strategy as part of the Option A combined dispatch authorised by PO chat 2026-05-04 «1. А»):**

1. Mandate at least one **boot smoke test** that builds the container, starts the process, and asserts a runtime-level success signal (e.g. structured log line `[gateway] ready` AND a mocked Telegram update reaches a real handler call). This test belongs in `tests/deployment/` (extending the existing TKT-013 `compose.test.ts` pattern) or in a new `tests/integration/` directory.
2. Update `docs/prompts/ticket-orchestrator.md` § cross-reviewer audit checklist to include «Did at least one test in this Ticket scope start the application process?» as a gate item for any TKT touching `src/main.ts`, `src/index.ts`, `Dockerfile`, `docker-compose.yml`, `src/deployment/**`, or `src/telegram/entrypoint.ts`.
3. Update `docs/meta/devin-session-handoff.md` § cross-reviewer ratification audit checklist to mirror the TO checklist addition.
4. Update `docs/meta/po-self-testing-guide.md` §1 to reference an `npm run smoke` (or equivalent) shortcut that PO can execute pre-deploy on `main` HEAD.
