# Question 01 for TKT-002@0.1.0

## Context

During iter 2 self-check, `git diff --name-only origin/main..HEAD` includes the RV-CODE-002 review artifact under `docs/reviews/` as a deletion because `origin/main` has one commit not present in this branch. The invocation requires that diff to show only TKT-002@0.1.0 §5 outputs plus the Ticket file, but the allowed fixes do not include touching `docs/reviews/`.

## Question

Which recovery path is authorized: A) allow a non-force merge commit from `origin/main` even though it makes `origin/main..HEAD` show 15 commits, B) authorize a force-with-lease rebase of the existing remote branch onto `origin/main`, or C) treat the `docs/reviews/` deletion as known base drift and push the current 14 commits unchanged?

## What I assumed / would do if no answer

Stop before push. I will not silently merge, rebase, force-push, or restore a `docs/reviews/` file because each option violates at least one explicit iter 2 constraint.

## Architect's answer

Option A is authorized. Merge commit is preferred over rebase
(Option B / force-with-lease) because:
  1. Rebase rewrites commit hashes; PR #12's review history (Devin
     Review comments, Reviewer-LLM PR #13 review trail) is
     hash-anchored. Rewriting hashes would orphan that audit trail.
  2. The "15 commits" cosmetic concern in the question is not a
     constraint anywhere in CONTRIBUTING.md, executor.md, or
     handoff §5. The iter 2 invocation's count rule was a
     scope-drift detection mechanism, not a hard cap.
  3. Option C (push as-is) leaves the diff perpetually ambiguous
     and forces every future reviewer to mentally subtract the
     deletion. That is a process smell; we resolve it now.
After the merge commit, `docs/reviews/RV-CODE-002-...md` will be
present on both `origin/main` and HEAD, so it disappears from the
diff. The diff will then show only TKT-002 §5 outputs + the Ticket
file + docs/questions/Q-TKT-002-01.md (which is allowed via the
Question Protocol carve-out).
